Background

What the Mira API does

Mira API extends Mira AI's news and social intelligence into a customer's own AI assistants, agents, and internal tools — the same engine behind Mira Studio, accessible programmatically instead of through the Meltwater UI.

This is a paid add-on, not a bundled feature. The Mira API is sold separately from the standard Meltwater Insights API. A customer having one does not mean they have the other — confirm access before you assume it, and don't promise activation to a customer who hasn't purchased it. Access is tiered: Trial (500 monthly prompts, 90 days) through Plus, Advanced, Premium, and Elite (up to 40,000 monthly prompts). This guide assumes the customer already has access at one of these tiers — if they don't, loop in their Account Manager for a trial or purchase conversation rather than trying to activate anything yourself.
💬

Conversational, AI-generated insight

Customers (or their internal tools) ask plain-language questions and get back a written, synthesized answer — not raw mentions, but an AI-generated summary grounded in Meltwater's data.

🔌

Built for embedding, not browsing

Designed to be called from a customer's own chatbot, internal AI assistant, or dashboard — not a UI a person logs into. The value is Mira's intelligence showing up inside tools the customer already uses.

🧵

Multi-turn, with memory

Supports threaded conversations — a follow-up question can reference everything asked before it in that thread, up to a long context window, without the customer re-explaining themselves.

📎

Cited, not just generated

Responses come back with citations back to the underlying coverage, so outputs are traceable — not a black-box answer, but one a customer can verify and defend internally.

Mental model for IMs: If a customer already uses Mira Studio inside the Meltwater app and likes it, the Mira API is "that, but callable from your own tools instead of from a login screen." Same engine, same quality of output — different delivery mechanism. If they've never used Mira Studio, walking them through it first (in the app) is often the fastest way to show them what the API would deliver programmatically.
Know this before any conversation

Mira API, Meltwater API, and Meltwater MCP aren't the same thing

Customers will use these names loosely. Knowing the difference — fast — is the single most important thing in this guide. Getting this wrong in front of a customer (or promising something this product doesn't do) is the most common mistake to avoid.

Meltwater API

Content and analytics

Structured, mention-based data — articles, posts, metrics, sentiment scores. Built for pipelines, BI tools, dashboards, and data warehouses. The customer's system does the analysis; the API just delivers the raw structured data. This is covered in the separate Meltwater API guide.

Mira API

Chat-based AI insight (this guide)

Conversational, AI-generated responses powered by Mira AI — the same outputs Mira Studio produces inside the app, delivered via a chat-completion API instead. Built for embedding into a customer's own chatbots, internal agents, or executive briefing tools. Exposes two tools when accessed via MCP: ask a question, and list projects. Mira Studio outputs only — nothing broader.

Meltwater MCP

The full agentic tool layer — separate product

A much broader product: full read/write access to searches and assets, journalist and influencer data, AI visibility, and more — callable from Claude, ChatGPT, or any MCP-compatible AI host. This is not the Mira API and not what this guide covers. Pricing is still TBD and it isn't sold standalone today. If a customer is describing this, see the objection below — loop in the Account Manager.

The shorthand that holds up under questioning: Meltwater API gets data out. Mira API lets a customer ask Mira AI a question and get a conversational answer back. Meltwater MCP is the delivery layer that makes Meltwater callable as a tool inside whatever AI environment a customer already works in — a much bigger surface area than Mira API's "ask + list projects." Think of Mira API as a single gate; Meltwater MCP is the whole airport.

↓ Start with discovery
Path 2 · Step 1

Discovery questions

Ask these before explaining anything about the API itself. The goal is finding a real, specific moment in their workflow where an AI-generated answer — not raw data — would save time or look impressive to leadership.

1
When your team needs a quick read on "what's happening with our brand right now," what does that process look like today? Listen for: manual digging through Explore, someone compiling a summary by hand, or "we just don't have a fast answer for that." Any of these is a Mira API opening.
2
Does your team use any internal AI tools or assistants today — Claude, ChatGPT, a custom internal bot? Listen for: yes to any of these. That's an existing surface where Mira API output could show up without building anything new from scratch.
3
Who outside of the people who log into Meltwater would benefit from getting quick media or social insight — leadership, a regional team, an exec? Listen for: any name or function. This is the "expand beyond traditional users" angle — Mira API can put insight in front of people who'd never log into the platform.
4
Do you have a developer or data/AI team that's already building internal tools or chatbots? Listen for: yes. This is the technical stakeholder who'd actually implement the integration — get them on a call before going further down this path.
5
How often does someone on your team have to turn raw coverage into a written summary, brief, or report for someone else? Listen for: a recurring cadence (daily briefs, weekly reports, campaign recaps). This is the clearest "minutes instead of hours" pitch, and it mirrors what Mira Studio already does inside the app.

Path 2 · Step 3

Common objections

For customers who push back, go quiet, or ask about something this product doesn't do. Keep responses grounded and don't overpromise — when in doubt, route to the AM rather than committing on the spot.

"Can you just connect Meltwater to our Claude/ChatGPT setup the way [a competitor / something we saw] does — full access, not just chat answers?"
What you're describing sounds closer to Meltwater MCP, which is a separate, broader product from the Mira API — it exposes a much wider set of tools (search and asset management, journalist and influencer data, AI visibility) versus Mira API's ask-a-question-and-get-an-answer model. I can speak to it at a high level, but pricing and packaging for that are still being finalized, so I'd want to loop in your Account Manager before we go further down that road.
"We don't have a developer to build this."
That's worth knowing upfront — Mira API is built for integration into a customer's own tools, so some technical lift is real. If you don't have someone in-house, this might be a "park it for later" conversation rather than an active build. I'd still recommend confirming who'd own it if priorities shift, so we're not starting from zero later.
"Could we use this to train our own AI model on Meltwater's content?"
No — that's explicitly outside what's supported. Providing a full feed of raw article text to train large language models isn't permitted under Meltwater's data licensing; only summarized or insight-based content delivery is allowed. If they want to explore something AI-related beyond that, loop in the Account Manager rather than improvising an answer here.
"Could we build a product on top of this and sell it to our own customers?"
No — Meltwater data and insights can't be integrated into a product intended for resale. That's a usage policy violation regardless of how the integration is built. This is worth flagging early if a customer mentions building anything customer-facing on top of Mira API.
"We're already overwhelmed getting the core platform set up."
Totally reasonable — let's keep the focus on fundamentals for now. I'll note Mira API as a phase-two item in our plan and bring it back up once things are stable. There's no urgency on activation timing if the access is already part of the package.
Final step · Both paths

Activation & integration

Getting the token activated is the easy part — done in two business days, no customer effort required. The part that actually determines whether a customer adopts this is what happens in the weeks after that. This section covers what good onboarding looks like, the real mechanics of connecting to Claude and OpenAI, starter prompts, and how Claude Skills fit in on top of the connection.

Your first move: confirm the customer's contract includes a Mira API tier (Trial, Plus, Advanced, Premium, or Elite) — it does not come bundled with standard Meltwater API access. If you're not sure, check with the Account Manager before submitting anything.
1

Submit the activation request (you)

Email help@meltwater.com to request Mira API activation for the customer's account. Include:

• The customer's account name and confirmed Mira API tier
• The name and email of the designated API User who will manage the token
• A link to the customer's Salesforce Opportunity

⚠️ Confirm the API User with the customer before submitting — it should be whoever owns the technical implementation, not just the primary contact.
2

Wait for Support to enable access

Meltwater Support enables Mira API access on the account after receiving your ticket. No customer action needed at this stage.

⏱ Standard turnaround is 2 business days. Set that expectation with the customer upfront.
3

Customer creates their API token

Once Support confirms access is on, the designated API User creates their token in the Meltwater app under Account > Meltwater API — same flow as standard Meltwater API access. The token is shown once at creation, so it must be copied and stored securely immediately. Full steps: API Credentials guide

What good looks like

A real Mira API onboarding journey

Token creation isn't the finish line — it's the starting gun. Customers who get a working integration in front of real users within the first couple of weeks stick. Customers who get a token and nothing else go dormant. This is the sequence to actually drive toward.

Week 1

Get the developer talking to the API directly

Before any Claude or OpenAI wiring, have the customer's developer make one successful call to the Responses endpoint — a single curl request is enough to prove the token works and show them the shape of a real response, citations included.

  • Confirms the token and account are live before more complex integration work starts
  • Gives the developer a real example response to design around, instead of working from docs alone
  • Takes minutes, not days — a good "win #1" to report back to the customer's stakeholders
Week 1–2

Connect it to wherever the team already works

Identify whether the team is Claude-based, OpenAI-based, or building something fully custom, and wire up that specific path — see the integration guidance below. This is almost always the developer's task, but you should know enough to sanity-check progress and unblock common snags.

  • If they're a Claude Desktop shop: get the MCP server configured and have one person ask Mira a real question inside Claude
  • If they're OpenAI-based: get one working call through the Responses API with Mira as a tool
  • Either way — the goal for this stage is one person, one real question, one real answer. Not a finished product.
Week 2–3

Build the first real recurring use case

Take whatever came up in discovery — a daily brief, a weekly report, a Slack digest — and get a working version of that specific thing in front of the person who'll actually use it. Use the starter prompts below as a basis if the customer hasn't written their own yet.

  • This is the moment to introduce Claude Skills if they're a Claude shop — see below
  • Push for a real recipient, not a test account — adoption sticks when a real stakeholder sees value, not when a developer confirms it technically works
Week 3–4

Check in, expand, and hand off ownership

Confirm the integration is actually being used — not just built. Ask what's working, what's missing, and whether there's a second use case worth tackling. By this point the customer's team should own the integration; your job shifts to occasional support rather than active building.

  • If usage is low, the most common cause is that no one told the intended recipient it exists — close that loop directly
  • If usage is strong, this is the moment to flag a tier upgrade conversation with the Account Manager if prompt volume is climbing

Helping customers connect Mira API to Claude & OpenAI

How the integration actually works

You're not writing this code with the customer — that's their developer's job. But knowing the actual shape of the setup means you can sanity-check progress, unblock a stuck conversation, and point to the exact doc page instead of "go check developer.meltwater.com" and hoping they find it.

◆
Connecting to Claude Desktop
MCP server

Mira API is exposed as a remote MCP server. The developer edits Claude Desktop's config file directly and adds Meltwater as an MCP server, authenticating with the Mira API token as a header.

claude_desktop_config.json
{
  "mcpServers": {
    "meltwater": {
      "command": "npx",
      "args": [
        "-y",
        "mcp-remote",
        "https://api.meltwater.com/mcp",
        "--header",
        "apikey: ${MELTWATER_API_KEY}"
      ],
      "env": {
        "MELTWATER_API_KEY": "<your api key>"
      }
    }
  }
}

File lives at ~/Library/Application Support/Claude/claude_desktop_config.json on macOS or %APPDATA%\Claude\claude_desktop_config.json on Windows — or just Settings > Developer > Edit Config inside Claude Desktop. Save, restart Claude Desktop, and Meltwater shows up as an available tool.

Connecting to OpenAI
Responses API

The same MCP server can be registered as a tool inside OpenAI's Responses API. The developer adds Meltwater's MCP endpoint as a tool source and the model decides when to call it.

python — registering Meltwater as a tool
from openai import OpenAI

client = OpenAI(api_key="<YOUR OPENAI API KEY>")

resp = client.responses.create(
    model="gpt-4.1",
    tools=[{
        "type": "mcp",
        "server_label": "meltwater-api",
        "server_url": "https://api.meltwater.com/mcp",
        "require_approval": "never",
        "headers": {
            "apikey": "<YOUR MELTWATER API KEY>"
        }
    }],
    input="What is going on in the news with fast fashion in the last 7 days?",
)
print(resp.output_text)

For a custom build without MCP at all, a developer can call the chat-completion endpoint directly — POST /v3/mira/responses — and wire the response into whatever workflow they're building themselves.

Starter prompts worth handing to a customer

These work whether the customer is testing the raw API, asking through Claude, or asking through an OpenAI-powered tool — the prompt is what changes, not the integration. Good Mira prompts are specific and time-bound, same as the discovery principle for any AI-powered Meltwater product.

📋 Daily / weekly brief
"What are the key conversations around [Brand] in the last 7 days?"
"Summarize the top stories and sentiment shift for [Brand] this week."
⚔️ Competitive intel
"Compare sentiment and share of voice for [Brand] vs [Competitor] in the last month."
"What's the latest news or buzz around [Competitor]?"
🚨 Risk & issue monitoring
"What are the risks and opportunities in coverage of [Brand] this week?"
"Has there been any negative coverage of [Brand] in the last 48 hours? Summarize it."
📰 Campaign & coverage reports
"Create a coverage report for [Campaign] in the last 7 days."
"Which journalists covered [Campaign], and should we follow up with any of them?"
💡 These are the same prompt patterns that work in Mira Studio inside the app — that's intentional. If a customer already has prompts they like in Mira Studio, those same prompts are the fastest starting point for their API or Claude integration.
For Claude-based customers

Building a Claude Skill around Mira API

This is a layer on top of the MCP connection, not a separate integration. Worth understanding the distinction so you don't conflate the two with a customer.

MCP connection (the access)

Gets Mira API into Claude as a callable tool. This is the claude_desktop_config.json setup above. Without this, Claude has no way to reach Meltwater at all.

+

Claude Skill (the recipe)

A reusable set of instructions — a SKILL.md file — that tells Claude exactly how to use that connection for a specific recurring task, so the customer doesn't re-explain the format every time.

In plain terms: MCP is what lets Claude reach Meltwater at all. A Skill is what teaches Claude to do a specific job with that connection the same way every time — without the customer re-typing instructions in every conversation.

SKILL.md — example: weekly leadership brief

What it would tell Claude to do: "When asked for the weekly brief, call Mira API for [Brand]'s coverage from the last 7 days, including sentiment and top sources. Format the response as: headline summary, 3 key stories, sentiment trend line, and one flagged risk or opportunity. Keep it under 300 words. Always include citations."

Result: anyone on the customer's team can ask Claude for "this week's brief" and get the same consistently-formatted output every time — without retyping the formatting instructions, and without needing to know how to write a good Mira prompt themselves.

💡 This is the natural next step after the Week 2–3 stage in the onboarding journey above. If a customer has a recurring Mira API use case that's working but inconsistent in format, a Skill is usually the fix — not a more complicated prompt.

Claude Skills are a Claude feature, not a Meltwater one — they work with any MCP connection, not just Mira API. There's no Meltwater-specific setup needed beyond having the MCP connection live. Point technical customers to Anthropic's own documentation for how to author a Skill.

⚠ What Mira API does NOT support
  • It is not the full Meltwater MCP. Mira API's MCP exposure is limited to two tools — asking a question and listing projects. It does not support search/asset management, journalist and influencer data, or AI visibility tools. If a customer expects that broader toolset, they're thinking of Meltwater MCP, a separate product.
  • No training data exports. Providing a full feed of raw article text to train a customer's own LLM is not permitted — only summarized, insight-based output is allowed.
  • No resale or productization. Meltwater insights delivered through Mira API cannot be built into a product the customer sells to their own customers.
  • Not a replacement for the Meltwater API. If a customer needs raw structured mentions, metrics, or bulk export for a data warehouse or BI tool, that's the standard Meltwater Insights API — Mira API only returns conversational, AI-generated answers.
  • Mira Studio outputs only. The API returns the same kind of answer Mira Studio gives inside the app — it doesn't expose any capability beyond what Mira Studio itself can do.
