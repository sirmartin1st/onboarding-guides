Background

What the Meltwater API does

The API is included in certain Meltwater package tiers. Some customers will know it's there. Many won't. Either way, your job is to make sure they understand what they have and leave onboarding using it.

📤

Content export

Export media articles and social mentions that match their Meltwater searches — on a schedule or one-off for research projects.

📊

Metrics & analytics

Pull analysis, trend data, and topic-level analytics directly into any reporting workflow — without manual platform exports.

🔗

Connect internal tools

Feed Meltwater data into Tableau, Power BI, Domo, Looker Studio, or any tool that accepts CSV or JSON.

⚙️

Build automated workflows

Programmatically manage searches, trigger exports, and route signals into internal systems — no manual steps required.

Data use restriction: API data is licensed for internal use only. Customers cannot use it to build or resell commercial software products. X/Twitter content has additional platform-specific limitations. Direct them to developer.meltwater.com for details.

Choose your path

Before anything else, figure out where the customer stands. This determines everything about how you run the API portion of onboarding.

✅

They know they have it and want to use it

No education needed. Skip straight to activation — get the internal ticket submitted and get them live.

Signals: customer mentioned the API at kickoff, their technical team is already asking questions, they purchased it as a deliberate add-on.

↓ Jump to activation
💡

They don't know they have it — or aren't sure they need it

Start with discovery questions to understand their workflow, then use the talk tracks to connect the dots. Handle any objections, then move to activation.

Signals: customer hasn't mentioned the API, they're new to Meltwater, they're a package-included customer with no technical contact listed.

↓ Start with discovery
Path B · Step 1

Discovery questions

Ask these before you explain anything about the API. The goal is to understand how they currently work with data so you can connect the API to something real and specific for them.

1
What are you planning to do with your Meltwater data after you pull it? Listen for: manual exports, copy-pasting into reports, emailing CSVs around. Any of these signals a workflow the API can improve.
2
Do you use an external reporting tool — like Tableau, Power BI, Domo, or Looker? Listen for: yes to any BI tool. This is the clearest API use case. Meltwater has off-the-shelf connectors for several of these.
3
Is there anyone else on your team — an analyst, a developer, a data team — who owns your company's reporting infrastructure? Listen for: yes. This signals a technical stakeholder who may want to pull Meltwater data into a central data warehouse or dashboard. Get that person on a call.
4
How often are you building reports from Meltwater data — and how long does that process take? Listen for: weekly cadence or more, with more than a few minutes of manual work involved. A scheduled API export can eliminate that entirely.
5
Are there internal systems — a CRM, an internal dashboard, a data warehouse — where having Meltwater data automatically would be useful? Listen for: any mention of wanting data "piped" somewhere or consolidated with other business data. That's the API's core value in an enterprise context.
6
If Meltwater data could land in the tools your team already uses — automatically, on a schedule — what would that change about how you work? Let them describe the ideal state. Then you can show them the API is what gets them there. Good question to end on before transitioning to the talk track.
Path B · Step 2

Talk tracks

Use these after discovery — once you know enough about their workflow to make the conversation specific. Pick the track that matches what you heard.

Opening — introducing the API to a customer who doesn't know they have it

"One thing I want to make sure we cover — your package actually includes API access to Meltwater. It's something a lot of customers don't realize upfront, and it can be one of the most impactful parts of your subscription depending on how your team works with data."

"It lets you get your Meltwater data — mentions, analytics, trends — delivered directly into the tools your team already uses, without anyone having to log in and run exports manually. Based on what you've shared about your workflow, I think there's a clear fit here. Let me walk you through it."

If they use a BI tool (Tableau, Power BI, Domo, Looker)

"Since you're already working in [tool], the API connects directly. We have integration connectors that feed your Meltwater data straight into your existing dashboards — so instead of exporting a CSV and uploading it manually, your data flows in automatically on whatever cadence makes sense for your team."

"For teams that already have a BI setup, this is usually the fastest win with the API — fresher data, less manual work, and it lives right alongside everything else you're already reporting on."

If they have a technical team or developer

"If you have a developer or data analyst involved, the API will be familiar territory for them — it's a standard REST API with solid documentation at developer.meltwater.com. The typical setup is either a scheduled export that matches your reporting cadence, or a stream if your team needs data closer to real-time."

"A lot of our customers in this situation use the API to pull Meltwater data into a central data warehouse and combine it with internal metrics — so you can actually connect media coverage and social signals to pipeline, revenue, or other business data."

If they're handling everything manually today

"Based on what you've described, pulling data from Meltwater today involves some manual steps — logging in, running exports, moving files around. The API is designed to replace that."

"Once it's configured, your reporting process runs on a schedule without anyone touching it. The data goes where it needs to go automatically. Your team spends their time on the analysis, not the extraction."

Path B · Step 3

Common objections

For customers who push back or go quiet after the talk track. Keep responses grounded — the goal is to find the right fit or note it clearly for later.

"We don't have a developer to set this up."
That's worth knowing upfront. If you're using a BI tool like Tableau or Power BI, the integration connectors are built to be accessible to an analyst — not just a developer. For more custom setups, the technical lift is real, and it's worth looping in whoever on your team handles data infrastructure before deciding how to proceed. The documentation at developer.meltwater.com is a good starting point for them to assess the effort.
"We're not ready for this right now."
Completely fine — we don't need to activate it today. What I'd ask is that we identify who your API user will be now, so when the time comes, we're not starting from scratch. I'll note it in our onboarding plan and bring it back up when the core setup is stable. It's already part of your subscription, so there's no extra step to get access when you're ready.
"I'm not sure what we'd actually use it for."
That's a fair answer, and it usually means we need to dig a bit deeper into your current workflow. Can you walk me through how your team gets data out of Meltwater today and where it ends up? Once I understand that, I can tell you pretty quickly whether the API solves a real problem for you or whether it's something to revisit down the road.
"We're already overwhelmed getting the core platform set up."
Totally reasonable — let's keep the focus on the fundamentals for now. I'll note the API as a phase two goal in our plan and flag it when we're ready to revisit. The main thing I want to do today is confirm who on your team would own it when the time comes, so we have the right person in the loop.
Final step · Both paths

Activation

Whether you got here directly or after a discovery conversation, the activation process is the same. Submit the internal ticket, wait for Support to confirm, then send the customer email.

Your first move: API access has to be enabled on the customer's account by Meltwater Support before the customer can do anything. Submit the internal ticket as soon as the customer confirms they want to proceed.
1

Submit the activation request (you)

Email help@meltwater.com to request API activation for the customer's account. Your request should include:

• The customer's account name
• The name and email of the designated API User — the person who will manage tokens and own the integration on their end
• A link to the customer's Salesforce Opportunity

⚠️ Only one user per account can be designated as the API User. Confirm this with the customer before you submit — it should be the person doing the technical implementation, not just the primary contact.
2

Wait for Support to enable access

Meltwater Support will enable API access on the account after receiving your ticket. No customer action needed at this stage.

⏱ Standard turnaround is 2 business days. Set that expectation with the customer upfront, and follow up with Support if you haven't heard back.
3

Send the customer email

Once Support confirms access is on, send the email template below to the designated API User. It gives them everything they need to create their first token and get started independently.
